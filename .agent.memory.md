## 文件夹摘要

> 描述本文件夹在工程中的位置、定位与作用。

项目根目录。这是一个**具身家庭智能体**，运行于树莓派 5 + 机器车上，以人体神经系统为架构蓝本设计。不是问答式助手，而是能持续感知环境、主动判断并自主行动的 Agent。

整体架构分为两大运行环境：
- **Platform 层**（Python FastAPI，端口 8001）：硬件抽象层，管理所有需要 Python 才能访问的硬件（麦克风、扬声器、电机、Roborock SDK），通过 WebSocket `/ws` 与 Node.js 双向通信
- **Agent 层**（Node.js/Next.js）：神经系统核心，包含 runtime（Spine 事件总线 + PlatformConnector）、perception（感知层）、cognition（认知层）、behavior（行为层）

## 遗留问题

> 记录暂时性妥协、技术债或未完成的优化。

- 音频模块（麦克风 + 扬声器）已于硬件接入后启用（`platform/main.py` 中 AudioSensor / AudioEffector 已取消注释），验证入口为 `POST /audio/verify`
- 超声波模块（HC-SR04）已完成硬件接入（`platform/devices/ultrasonic.py` + `/ultrasonic/*` 接口 + `sense.ultrasonic.too_close` 事件），Node.js Agent 工具层尚未封装该能力
- `speak_end` 事件检测方式（`speaker._queue.empty()`）存在竞态：若 LLM 多步生成句子时第一句播完时第二句尚未入队，会提前触发 speak_end 进入 LISTENING；实际影响有限（用户仍有 ~10s 窗口继续说话），暂不修复
- 扬声器播放采用软件增益（`SOFTWARE_GAIN=2.0`）补偿 USB 声卡输出偏小，若出现破音可在 `platform/devices/speaker.py` 顶部降低此值；或用 `alsamixer` 调节硬件音量（更稳定）
- 麦克风 VAD 回调块大小（`BLOCK_DURATION_MS=300ms`）是为抑制 Platform 多任务 GIL 争抢下的 input overflow 而设置，可能引入最多 300ms 的语音端点检测延迟
- SLAM 建图模块已实现（`platform/slam/`，breezyslam），支持 REST 操作（start/stop/save/load）和 WebSocket 实时广播（pose + map PNG），但 Node.js Agent 工具层尚未封装，LLM 暂无法主动触发建图
- 地图分区（手动标注 Web UI）、A* 全局路径规划、DWA 局部避障尚未实现，是下一阶段工作
- 摄像头型号（小白 CMSXJ60A）对应的 miIO token 尚未获取，摄像头云台转向功能未实现，目前仅支持 RTSP 截图
- `getCleaningHistory` 工具读取本地 SQLite，而 Platform 的 `/history` 接口从 Roborock 云端拉取，两个数据源未打通，存在不一致风险
- 对话历史（`conversations` 表）与线程消息（`thread_messages` 表）存在功能重叠，`conversations` 表目前实际未使用
- `perception/chat/` 模块骨架已创建（notifyChatInput / notifyChatComplete），但尚未集成到 `app/api/chat/route.ts`，待接入后可让 ConversationManager 感知 Web Chat 活动

## 特殊决策

> 记录非标准实现的原因、用户指定的特殊要求（含来源）。

- **双层架构（Node.js + Python Platform）**：python-roborock SDK 仅提供 Python 版本，因此用 FastAPI 桥接，Node.js Agent 通过 HTTP 调用，两层各司其职
- **视觉分析使用通义千问 Qwen-VL**（非 OpenAI Vision）：阿里云 Dashscope，模型为 `qwen-vl-max`，通过 OpenAI 兼容接口接入
- **AI Agent 模型**：在 `src/core/cognition/brain/index.ts` 中通过 `AGENT_MODEL` 常量配置，与视觉模型分开管理
- **神经系统架构四层命名**（用户决策）：runtime（物理基底）/ perception（感知层）/ cognition（认知层）/ behavior（行为层）。perception 为多通道设计：audio（语音）和 chat（文字）都是感知入口，Brain 不区分来源
- **`lib/` 保留用于通用基础设施**：`db/` 和 `utils.ts` 无业务属性，放 `lib/`；Agent 神经系统逻辑全部在 `core/`
- **`agent/` 目录重命名为 `tools/`**：原 `agent/` 只是工具注册表，整个系统本身才是 agent，避免命名歧义
- **Sensory 内部消化原始流，Spine 只传离散事件**：连续 PCM 流在 AudioSensor 内部经 VAD 缓冲，只将完整语音块作为 `sense.audio.speech_end` 事件发布
- **Platform ↔ Node.js 通信使用 WebSocket**：原有 REST 是单向轮询，不满足实时音频事件推送需求；改为 Platform 暴露 `/ws` 端点，PlatformConnector 作为客户端长连接

## 注意事项

> 记录隐式约定、相似功能函数的区分与选择指南。

- **多轮对话流程**：唤醒词 → THINKING → LLM + 工具调用 → SPEAKING（TTS 播放）→ Platform 发 `sense.audio.speak_end` → Node.js manager 收到后 → LISTENING（10s 窗口，用户无需再说唤醒词）→ 超时 → IDLE
- **`sense.audio.speak_end` 传播链**：`Speaker._do_speak() finally → AudioEffector._on_play_end() → ws_manager.broadcast() → PlatformConnector._handleInbound() → INBOUND_MAP 映射 → Spine.publish → ConversationManager 订阅`。任一环节断裂都会导致 SPEAKING 卡 30s 后才回 IDLE
- **语音工具调用**：`generateVoiceResponse` 接受可选 `tools` 参数（由 `manager.ts` 注入 `ALL_TOOLS`），使 LLM 在语音模式下也能调用运动/拍照/扫地等工具；若 LLM 仅调用工具未生成口语输出，manager 直接回 IDLE
- **`unmute()` 副作用**：调用时会同时重置 VAD 的 `_is_speaking` 状态，丢弃静音期间积累的脏缓冲，防止 TTS 回声被当作用户语音发给 STT
- Platform 服务默认端口 **8001**，HTTP 通过环境变量 `PLATFORM_URL` 覆盖，WebSocket 通过 `PLATFORM_WS_URL` 覆盖
- 摄像头 RTSP 地址通过环境变量 `CAMERA_RTSP_URL` 配置，视觉分析 API Key 为 `DASHSCOPE_API_KEY`
- 数据库文件位于 `data/home-agent.db`，使用 WAL 模式，Next.js 热重载时通过 `globalThis` 保持单例
- Platform 启动命令：`cd platform && uvicorn main:app --host 0.0.0.0 --port 8001 --reload`（或 `pnpm platform`）
- 前端启动命令：`pnpm dev`
