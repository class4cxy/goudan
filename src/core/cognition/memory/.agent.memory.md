## 文件夹摘要

> 描述本文件夹在工程中的位置、定位与作用。

`src/lib/memory/` 是 Agent 的记忆管理层，负责将无限增长的对话历史通过分级压缩算法转化为线性权重衰减的上下文摘要，实现"越新权重越高、越旧信息越精简"的有损记忆机制。

## 文件夹内容结构

> 描述主要文件/子目录的功能、关键类或函数。

- **`conversation-buffer.ts`**：核心类 `ConversationBuffer`，渠道无关。
  - `getRawTail(messages)` — 返回未被任何 chunk 覆盖的最新消息（同步）
  - `assembleHistoryContext()` — 将 chunks 组装为注入 system prompt 的字符串，应用线性权重分配 token 预算
  - `maybeCompress(messages)` — 检查并触发压缩（async，建议 fire-and-forget 调用）
  - `getChunks()` — 返回当前 thread 所有 chunks（同步）

**关键常量（定义在文件顶部）：**

| 常量 | 默认值 | 含义 |
|---|---|---|
| `RAW_TAIL_SIZE` | 20 | rawTail 保留消息数，超出触发压缩 |
| `FLUSH_BATCH_SIZE` | 10 | 每次从 rawTail 刷出的消息批量 |
| `MERGE_THRESHOLD` | 5 | 同 level 累积多少 chunks 时向上合并 |
| `MAX_LEVEL` | 3 | 最高压缩代数 |
| `HISTORY_TOKEN_BUDGET` | 1500 | 历史摘要注入 prompt 的总 token 预算 |

**压缩层级示意：**
```
level 1 chunk：10条消息 → 1段摘要
level 2 chunk：50条消息 → 1段摘要（5个level-1合并）
level 3 chunk：250条消息 → 1段摘要（5个level-2合并）
```

**线性权重分配公式：**
- 共 n 个 chunk，第 i 个（从旧到新，i=1..n）权重 = i
- 总权重 = n*(n+1)/2
- 第 i 个 chunk token 预算 = (i / 总权重) × 1500

## 遗留问题

> 记录暂时性妥协、技术债或未完成的优化。

1. **压缩模型复用主模型**：`flushToChunk` 和 `maybeMergeChunks` 使用 `AGENT_MODEL`（deepseek-chat），理想情况应使用更廉价的小模型（如 deepseek-chat-lite 或 qwen-turbo）。待 API 配置扩展后替换。

2. **Token 计数为估算值**：`tokenCount` 用 `summary.length / 1.5` 粗估，不是真实 token 数。够用，但不精确。

3. **Voice / 微信渠道未接入**：当前只有 `/api/chat` 使用了 ConversationBuffer。Voice 渠道待 Voice Session 设计完成后接入，微信渠道尚未开发。

4. **`thread_messages` 存全量历史**：前端展示仍用完整消息，DB 行会随对话线性增长。长期需要考虑归档老历史或只存 rawTail + chunks。

## 特殊决策

> 记录非标准实现的原因、用户指定的特殊要求（含来源）。

1. **不修改 `thread_messages` 存储**（设计讨论 2026-03-09）：前端需要完整历史用于 UI 展示，ConversationBuffer 只影响传给 LLM 的上下文，不影响存储。因此 `getRawTail` 通过 `coveredCount()` 动态计算，而不是截断 DB 内容。

2. **渠道无关设计**（设计讨论 2026-03-09）：ConversationBuffer 只接受 `threadId`，不感知渠道类型。渠道差异（Voice 的 session timeout、微信的单线程）由上层决定何时创建/复用 threadId，压缩算法本身不变。

3. **线性权重衰减**（用户明确要求，2026-03-09）：权重随时间线性下降，而非指数衰减，保证即使很老的摘要也有少量 token 能进入上下文，避免完全遗忘。

4. **Fire-and-forget 压缩**：`maybeCompress` 在 `onFinish` 里非阻塞调用，避免压缩 LLM 调用延长用户等待时间。失败只记日志，下次对话时会再次检查并重试（幂等）。

## 注意事项

> 记录隐式约定、相似功能函数的区分与选择指南。

- `getRawTail()` 是**同步**的，可在请求路径上直接调用。
- `maybeCompress()` 是**异步**的，必须 fire-and-forget 或在 `onFinish` 里 await，不能在 `streamText` 调用前 await（否则会在对话完成前就触发压缩）。
- `assembleHistoryContext()` 返回空字符串（而非 null）当没有 chunks 时，调用方直接传给 `buildSystemPrompt` 即可，不需要判断。
- chunk 的 `covers_from / covers_to` 是消息的 **毫秒时间戳**（`UIMessage.createdAt` 转换而来），DB 的 `created_at` 是 `unixepoch()`（秒级），两者不同，注意区分。
