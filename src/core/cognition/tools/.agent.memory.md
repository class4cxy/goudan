## 文件夹摘要

> 描述本文件夹在工程中的位置、定位与作用。

AI Agent 核心层，负责模型配置、工具注册、Prompt 管理和定时任务执行。是 Node.js 侧所有 AI 能力的集中入口，由 `/api/chat` 路由和定时任务调度器共同使用。

## 文件夹内容结构

> 描述主要文件/子目录的功能、关键类或函数。

```
agent/
├── index.ts        # 核心入口：模型实例、ALL_TOOLS 注册表、executeScheduledTask
├── prompts.ts      # System Prompt 定义（buildSystemPrompt / SCHEDULER_PROMPT）
```

**关键导出：**

- `AGENT_MODEL`：DeepSeek `deepseek-chat` 模型实例，使用环境变量 `DEEPSEEK_API_KEY`
- `ALL_TOOLS`：所有可用工具的注册表 dict，传给 `streamText` / `generateText`
- `executeScheduledTask(task)`：定时任务执行入口，内部调用 `generateText` + `ALL_TOOLS`，最多 6 步

## 遗留问题

> 记录暂时性妥协、技术债或未完成的优化。

- Bridge 已有 `/clean/stop`（原地停止）和 `/clean/zone`（区域清扫）接口，但 `ALL_TOOLS` 中尚未对应封装工具
- 对话模式（`/api/chat`）使用 `streamText`，最多 **8 步**；定时任务模式（`executeScheduledTask`）使用 `generateText`，最多 **6 步**——两处步数上限不同，需注意
- SLAM 探索链路依赖 Platform 的 `/slam/start` 与传感器状态（LiDAR/超声波）；工具层需以 Explorer 实际运行状态作为启动成功判据，不能仅以 HTTP 请求发出即视为成功

## 特殊决策

> 记录非标准实现的原因、用户指定的特殊要求（含来源）。

- **主模型选用 DeepSeek**（非 OpenAI）：`deepseek-chat`，成本低；视觉分析单独使用 Qwen-VL，两个模型分工明确
- **定时任务执行与对话共用同一套工具**：`executeScheduledTask` 直接复用 `ALL_TOOLS`，Scheduler Prompt 通过 `SCHEDULER_PROMPT` 单独定制，无需维护两套工具
- **运动工具闭环优先与固定标定速度（2026-04）**：`motor.ts` 的 `navigateTo` 保留中文文本容错提取（如“前进1米”“右转90度”自动识别 `distance_mm/angle_deg`），并将闭环默认速度/上限统一为 `40`（`CLOSED_LOOP_DEFAULT_SPEED=40`、`CLOSED_LOOP_MAX_SPEED=40`）以匹配 Platform `motor/drive` 标定速度，避免“接口直调正常但 Agent 闭环几乎不动”。
- **开环安全闸（2026-03）**：`navigateTo` 对“前进/后退但未给距离”的请求，默认改为短时点动（`OPEN_LOOP_DEFAULT_DURATION_S`，默认 0.8s），只有文本明确含“持续/一直/连续”等词才允许持续运动。并新增 `CLOSED_LOOP_MAX_SPEED`（默认 35）限制闭环速度上限，降低高速打滑导致的距离失真。
- **闭环超时收敛（2026-04）**：`motor.ts` 增加 `CLOSED_LOOP_TIMEOUT_MAX_S` 与按距离/角度估算的超时参数（`CLOSED_LOOP_TIMEOUT_DISTANCE_PER_100MM_S` / `CLOSED_LOOP_TIMEOUT_ANGLE_PER_30DEG_S`）。默认将闭环超时上限收敛到 10s，避免里程计失真时“持续跑很久”。闭环失败信息现在包含 `traveled_mm/rotated_deg` 细节，便于快速判断是传感器计数偏低还是其他异常。
- **运动前等待音频空闲（2026-04）**：`navigateTo` 在执行任何电机动作前，先短暂轮询 Platform `/audio/status`，等待 `speaker.busy=false`。原因：实际链路中 LLM 常先播报“我来让机器车前进 1 米”，若 TTS 播放与底盘运动重叠，会让 Agent 运行环境与 `encoder_accuracy_test.py` 的纯运动测试环境不一致，导致距离漂移与抖动增大。
- **手动遥控优先权（2026-04）**：当 Platform `/teleop/status.enabled=true` 时，`navigateTo` 与 `navigateToCoordinates` 会拒绝自动运动请求（仅 `navigateTo` 的 `stop` 允许穿透），避免 AI 自动导航/动作与用户摇杆手控争抢底盘控制权。

## 注意事项

> 描述隐式约定、相似功能函数的区分与选择指南。

- 新增工具必须在 `ALL_TOOLS` 中注册，否则 Agent 无法调用
- `AGENT_MODEL` 只在此文件定义，对话路由和 Scheduler 都从这里 import，不要各自单独实例化
- `buildSystemPrompt()` 用于对话场景（含当前时间等动态信息），`SCHEDULER_PROMPT` 用于自动执行场景（更精简）
- `navigateTo` 中“直行无 distance_mm”是持续运动（开环）；若用户说了具体距离/角度，优先走 `/motor/drive` 闭环。部署时建议在环境变量显式设置 `CLOSED_LOOP_DEFAULT_SPEED`（推荐 20~35）。
- 开环模式默认受安全闸限制：未明确“持续运动”时仅短时点动，避免误触发长时间行驶；若业务需要旧行为，可在上层显式传入 `duration` 并使用“持续”表述。
- `navigateTo` 的语义导航当前只接受坐标字符串（`"x_mm,y_mm"`）；房间名（如“客厅”）会直接返回失败提示，不再返回“意图已记录”这种误导性成功结果。
