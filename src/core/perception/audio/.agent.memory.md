## 文件夹摘要

> 描述本文件夹在工程中的位置、定位与作用。

`src/lib/audio/` 是音频硬件适配的**丘脑层（Thalamus）+ 效应器（Effector）**，职责严格限定为：
- **感知侧**：将 Bridge 传来的 PCM 原始音频转换为语义事件（STT + 唤醒词检测 + 情绪分析）
- **输出侧**：将 Spine 的 `action.speak` 路由到 Bridge TTS

不含任何对话状态逻辑。对话状态机、队列管理、主动发起等全部在 `src/lib/conversation/` 中。

## 文件夹内容结构

> 描述主要文件/子目录的功能、关键类或函数。

```
audio/
├── thalamus.ts   # AudioThalamus：STT + 唤醒词检测 + 情绪分析 → 发布 Spine 事件
└── effector.ts   # AudioEffector：订阅 action.speak → BridgeConnector.send()
```

**thalamus.ts 数据流：**
```
sense.audio.speech_end（PCM base64）
    ↓
Whisper STT
    ↓
命中唤醒词？
  ├─ 是 → sense.audio.keyword（ConversationManager 处理）
  └─ 否 → sense.audio.transcript + sense.audio.emotion（Manager/AmbientAnalyzer 处理）
```

**唤醒词配置：** 环境变量 `WAKE_WORDS`，逗号分隔，默认 `Aria,小豆,狗蛋,aria`

## 遗留问题

> 记录暂时性妥协、技术债或未完成的优化。

- 情绪分析是基于关键词的简单规则，误报率较高；后续可接入小型情绪分类模型
- STT 使用 `whisper-1` 云端 API，有网络延迟（约 1-3s）；Pi 5 上可考虑本地 Whisper tiny
- `pcmToWav` 函数写在 thalamus.ts 内部，若其他模块需要可提取到 `utils/audio.ts`
- 唤醒词检测在 Whisper 之后（延迟 1-3s），未来可在 Bridge 侧接入 openWakeWord 实现 < 100ms 唤醒
- 转写文本过滤目前是规则法（口头噪声词/最短长度/强意图词），对边界场景仍可能误杀；后续可引入 ASR 置信度联合判断

## 特殊决策

> 记录非标准实现的原因、用户指定的特殊要求（含来源）。

- **audio/index.ts 已删除**：启动职责移交给 `src/lib/conversation/index.ts`（统一管理整个交流能力模块），`instrumentation.ts` 改为调用 `startConversationModule()`
- **唤醒词命中时不发 transcript**：命中唤醒词 → 只发 `sense.audio.keyword`，不再发 `sense.audio.transcript`，避免旁听分析器和对话管理器同时响应同一句话
- **AudioEffector 不直接操作硬件**：仅做 Spine → BridgeConnector 的路由，TTS 和播放在 Bridge Python 侧
- **STT 文本弱信号过滤**：`thalamus.ts` 在发布 transcript 前过滤口头噪声词（如“嗯/啊/是吗”）和极短弱意图文本，仅保留有唤醒词/强意图/足够信息量的转写

## 注意事项

> 记录隐式约定、相似功能函数的区分与选择指南。

- STT 需要 `OPENAI_API_KEY`；如果使用非 OpenAI 兼容接口，同时设置 `OPENAI_BASE_URL`
- 唤醒词环境变量：`WAKE_WORDS=小豆,Aria,aria`（逗号分隔，大小写不敏感）
- `sense.audio.keyword` 事件由 `ConversationManager` 直接订阅，不需要额外的 WakeListener 文件
- 本文件夹不持有任何状态，每个函数调用完全无副作用，所有状态在 `conversation/manager.ts` 中
- 文本过滤阈值可通过 `MIN_MEANINGFUL_TEXT_LEN` 调整（默认 2）
