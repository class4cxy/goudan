## 文件夹摘要

> 描述本文件夹在工程中的位置、定位与作用。

`src/lib/conversation/` 是 Home Agent 的**交流能力层**，对标人类的"主动/被动交流"能力。负责协调所有对话逻辑：状态机管理、优先队列调度、LLM 响应生成、旁听分析，以及主动发起机制。

与 `src/lib/audio/` 的分工：`audio/` 只做硬件适配（PCM→事件），`conversation/` 负责所有"说什么、什么时候说"的决策。

## 文件夹内容结构

> 描述主要文件/子目录的功能、关键类或函数。

```
conversation/
├── index.ts                    # 统一启动入口：startConversationModule()
├── manager.ts                  # ConversationManager（状态机 + 优先队列，核心）
├── context.ts                  # ConversationContext（语音对话历史，纯内存，滑动窗口）
├── brain.ts                    # generateVoiceResponse()：LLM 流式调用，按句回调
├── active/
│   ├── task-narrator.ts        # narrate() / narrateIssue()：任务层主动发言接口
│   ├── idle-initiator.ts       # 空闲超时定时器，随机策略发起闲聊
│   └── env-responder.ts        # 订阅 sense.audio.environment，环境声音主动响应
└── ambient/
    └── analyzer.ts             # 旁听分析：IDLE 状态下积累 transcript，定期 LLM 评分
```

**状态机：**
```
IDLE → LISTENING → THINKING → SPEAKING → IDLE
         ↑ 唤醒词（priority 0，任意状态立即中断）
```

**队列优先级：**
- priority 0：唤醒词，立即中断，清空队列
- priority 1：环境紧急（烟雾/哭声），中断 SPEAKING
- priority 2：任务问题/旁听兴趣，IDLE 时处理
- priority 3：空闲闲聊，IDLE 时处理，有过期时间

## 遗留问题

> 记录暂时性妥协、技术债或未完成的优化。

- `idle-initiator.ts` 中 `content: undefined` 的情景（让 LLM 自由发挥）尚未在 `manager.ts` 中处理：manager._speakActive 要求 content 非空，content 为空时走 `_speakActiveLLM`，triggerNote 可驱动 LLM 自由发挥，但若 triggerNote 也为空则 LLM 缺少上下文
- 旁听分析每次都独立调用 LLM 评分，若旁听内容频繁可能造成 API 浪费；后续可加入去重/节流（相同内容 hash 跳过）
- `env-responder.ts` 依赖的 `sense.audio.environment` 事件来自 Bridge YAMNet，YAMNet 尚未在 Bridge 侧集成
- `ConversationContext` 是纯内存，重启后历史丢失；语音对话历史暂不持久化（设计决策）

## 特殊决策

> 记录非标准实现的原因、用户指定的特殊要求（含来源）。

- **被动 vs 主动两种入口**：用户明确要求区分被动（唤醒触发）和主动（内部状态触发）两种对话场景，manager.ts 通过 `mode: 'passive' | 'active'` 区分，对应不同的起始状态（LISTENING vs SPEAKING）
- **ConversationManager 自己订阅 transcript**：不设独立的 WakeListener 文件，manager.start() 内部完成所有 Spine 订阅，保持状态机的内聚性
- **全局单例**：ConversationManager 通过 `globalThis.__conversationManager` 保持单例，Next.js 热重载安全
- **流式 TTS 逐句发布**：brain.ts 检测到句子边界就立即 publish action.speak，不等全文生成，首句延迟目标 < 1.5s
- **支持免唤醒调试模式**：新增 `VOICE_REQUIRE_WAKE_WORD`（默认 true）；设为 false 时，IDLE 状态下 `sense.audio.transcript` 可直接进入 THINKING，用于快速验证语音链路与本地模型

## 注意事项

> 记录隐式约定、相似功能函数的区分与选择指南。

- `narrate(text)` vs `narrateIssue(taskName, issue)`：前者直接说一句话，后者同时发布 `sense.agent.task_issue` 事件到 Spine（供其他模块感知），优先用 `narrateIssue` 报告任务异常
- `IdleInitiator.resetIdleTimer()` 已在 `manager.ts` 的 `_onWake()` 和 `_startThinking()` 中接入，每次对话活动自动重置空闲计时器
- `AmbientAnalyzer` 只在 `ConversationManager.getState() === 'IDLE'` 时处理 transcript，对话中不旁听，避免干扰
- `ConversationContext.reset()` 在每次唤醒时调用，开始新会话；不调用则多轮对话会延续上下文
- 若开启 `VOICE_REQUIRE_WAKE_WORD=false`，旁听分析触发会减少（IDLE transcript 会被主对话直接消费），仅建议用于调试或无唤醒词场景
