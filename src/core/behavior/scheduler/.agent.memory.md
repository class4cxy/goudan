## 文件夹摘要

> 描述本文件夹在工程中的位置、定位与作用。

定时任务引擎，基于 `node-cron` 实现。维护内存中的运行任务注册表（`runningJobs` Map），在 Next.js 服务启动时由 `providers.tsx` 初始化加载。任务执行时调用 Agent 层的 `executeScheduledTask` 完成自动化家务操作。

## 文件夹内容结构

> 描述主要文件/子目录的功能、关键类或函数。

**`index.ts` 导出：**

| 函数 | 说明 |
|------|------|
| `loadScheduledTasks()` | 从 DB 加载所有启用的任务并注册 cron job，服务启动时调用一次 |
| `scheduleTask(task)` | 将单个 ScheduledTask 注册到 node-cron，已存在则先停止旧 job |
| `unscheduleTask(taskId)` | 停止并移除内存中的 cron job |
| `addScheduledTask(name, cron, type, config)` | 写入 DB + 立即调度，返回新建的 ScheduledTask |
| `toggleTask(taskId, enabled)` | 启用/禁用任务（同步 DB 和内存 cron job） |
| `removeTask(taskId)` | 停止 cron job + 从 DB 删除 |
| `setTaskExecutor(fn)` | 注入任务执行器（`executeScheduledTask`），解耦 scheduler 与 agent 层 |

**内部状态：**
- `runningJobs: Map<number, cron.ScheduledTask>` — 内存中运行的 job 注册表
- `executor: TaskExecutor | null` — 注入的执行函数，调用前检查 `isQuietHour()`

## 遗留问题

> 记录暂时性妥协、技术债或未完成的优化。

- `runningJobs` 存于内存，Next.js 进程重启后会通过 `loadScheduledTasks()` 重新加载，但 **dev 热重载时可能出现重复注册**（hot reload 不完全等同于进程重启）
- 任务执行失败只打印 `console.error`，没有重试机制和失败记录存储

## 特殊决策

> 记录非标准实现的原因、用户指定的特殊要求（含来源）。

- **依赖注入 executor**：scheduler 通过 `setTaskExecutor` 接收 agent 层函数，避免循环依赖（scheduler → agent → db → scheduler）
- **静默时段跳过**：cron 触发后先调用 `isQuietHour()`，若在静默时段（默认 22:00-7:00）则跳过本次执行，不中止任务本身

## 注意事项

> 记录隐式约定、相似功能函数的区分与选择指南。

- `loadScheduledTasks()` 在 `src/components/providers.tsx` 的 `useEffect` 中调用，**仅服务端执行一次**（通过 `/api/scheduler/init` 或类似机制触发）
- `addScheduledTask` 和 `addScheduledTaskTool`（tools/scheduler-tool.ts）功能相同，前者是底层函数，后者是 AI 工具包装——Agent 调用 tool，API 路由调用底层函数
- 修改 `runningJobs` 状态的函数（`scheduleTask`/`unscheduleTask`）不是线程安全的，Node.js 单线程下一般无问题，但避免并发调用同一 taskId
